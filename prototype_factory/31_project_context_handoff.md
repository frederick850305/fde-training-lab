# 2-31 项目上下文流转打通

## 本节目标

本节在 2-30 的“原型工厂操作台总览”基础上，继续向真实工作流迈进一步：

先打通“操作台总览 -> 需求输入工作台”的第一条前台上下文流转链路。

当前仍然不调用 DeepSeek，也不调用 FastAPI，不引入新的 npm 依赖。本节不追求一次性把所有工作台都改成真实上下文驱动，而是先验证最小可用的“项目上下文”传递机制。

## 本节价值

前面已经做到：

- 第一个 Tab 有了“原型工厂操作台总览”
- 可以从总览点击进入对应工作台

但当前仍只是“页面跳转”，不是“带着上下文进入下一步”。

本节通过新增统一的前台项目上下文状态，先完成：

1. 在操作台总览中设置当前项目上下文
2. 进入需求输入工作台时展示当前上下文
3. 让需求输入页能够消费这份上下文信息

这一步是后续接入真实项目执行链路的前置基础。

## 当前交互规则

- `App.vue` 维护一个统一的 `projectContext`。
- 在操作台总览中，可注入当前演示项目上下文。
- 点击“进入需求输入”时，带着当前上下文进入需求输入工作台。
- 需求输入页展示当前项目上下文卡片。

## 后续扩展方向

后续可以继续扩展：

- 将需求输入结果继续写回项目上下文。
- 把项目上下文传给场景识别、功能设计、页面设计等工作台。
- 在上下文中增加版本号、执行状态、最近更新时间。
- 将前台上下文替换为后端返回的真实项目状态。

## 验收方式

执行前端构建：

```bash
cd frontend
npm run build
```

浏览器打开页面后确认：

- 操作台总览可以显示并更新当前项目上下文。
- 点击进入需求输入后，需求输入页能看到当前上下文卡片。
- 第一 Tab 已开始从“页面跳转”演进为“上下文驱动的工作流入口”。
- 第二阶段练习进度更新为 `2-31：项目上下文流转打通`。
