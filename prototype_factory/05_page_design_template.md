# 2-5 页面清单设计模板

本文件用于把 2-4 的功能模块设计，进一步转成 Vue3 前端原型的页面清单、导航结构和页面优先级。

核心原则：

```text
页面不是功能的堆叠，而是用户完成任务的工作空间。
```

## 一、本节目标

完成 2-4 后，我们已经知道一期原型要做哪些功能。

2-5 要进一步回答：

```text
这些功能应该放在哪些页面里？
每个页面服务哪个角色和场景？

哪些页面一期必须做？
哪些页面只是后续扩展？
```

这一步的输出会直接影响：

```text
Vue Router 路由设计

components 拆分
页面生成 Prompt

```

## 二、页面设计的基本思路

页面清单不要简单等同于功能模块清单。

一个页面可以承载多个功能：

```text
客户需求分析页
  -> 参数输入
  -> 任务提交
  -> loading 状态
  -> 成功结果
  -> 错误提示
```

一个功能也可能拆到多个页面或区域：

```text
报告结果展示
  -> 在需求分析页展示输出路径
  -> 后续可扩展为报告预览页
```

推荐设计顺序：

```text
用户角色
  -> 核心场景
  -> 任务流程
  -> 页面
  -> 页面区域
  -> 组件
```

## 三、页面类型划分

在业务原型中，常见页面可以分为以下类型。

| 页面类型 | 作用 | 示例 |
|---|---|---|
| 工作台页 | 展示系统能力、状态和入口 | 首页、控制台 |
| 业务操作页 | 支持用户完成核心任务 | 客户需求分析页 |
| 列表页 | 展示多条记录或任务 | 历史任务列表 |
| 详情页 | 展示单个对象的详细信息 | 报告详情页 |
| 看板页 | 展示指标、趋势、风险 | 运营看板 |
| 配置页 | 管理规则、字典、参数 | 系统配置页 |
| 方法资产页 | 展示 Prompt、模板、说明 | Prompt 模板页 |

第二阶段一期建议优先做：

```text
工作台页 + 核心业务操作页 + 简单状态页
```

## 五、页面优先级标准

页面优先级建议使用 P0/P1/P2/P3。

| 优先级 | 含义 | 判断标准 |
|---|---|---|
| P0 | 一期必须做 | 没有该页面，主流程无法演示 |
| P1 | 一期建议做 | 能明显增强演示完整度 |
| P2 | 后续扩展 | 有价值，但不影响一期闭环 |
| P3 | 暂不设计 | 当前阶段不需要，容易扩大范围 |

建议一期 P0 页面控制在：

```text
1-3 个
```

页面太多，会让原型开发变成“菜单很多但每页都浅”。

## 十、一期最小页面范围

如果时间有限，第二阶段最小页面范围建议是：

| 页面 | 是否必须 | 原因 |
|---|---|---|
| 工作台首页 | 必须 | 让用户理解系统定位和入口 |
| 客户需求分析页 | 必须 | 跑通第一条核心业务闭环 |
| 健康检查页 | 建议 | 便于确认后端是否可用 |
| Prompt 模板页 | 建议 | 体现 AI 原型工厂的方法资产特点 |

最小演示主线：

```text
打开工作台首页
  -> 进入客户需求分析页
  -> 输入需求文件路径
  -> 点击生成报告
  -> 页面展示成功结果或错误提示
  -> 回到 Prompt 模板页说明方法资产可复用
```

## 十一、页面设计质量检查

完成页面清单后，用下面清单检查。

| 检查项 | 合格标准 |
|---|---|
| 页面是否服务场景 | 每个 P0 页面都能追溯到核心场景 |
| 页面数量是否克制 | 一期 P0 页面不超过 3 个 |
| 页面目标是否明确 | 每个页面一句话能说明用途 |
| 导航是否简单 | 一级菜单即可支撑一期原型 |
| 状态是否完整 | 核心操作页有默认、loading、成功、失败状态 |
| 文件名是否清晰 | Vue 文件名能看出页面用途 |
| 是否能指导生成代码 | 页面区域和组件建议足够清楚 |
| 是否能支撑演示 | 页面顺序能组成 3 分钟演示路径 |
