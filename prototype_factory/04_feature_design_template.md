# 2-4 功能模块设计模板

本文件用于把 2-2 的需求拆解和 2-3 的业务场景，进一步转成可开发、可演示、可验收的功能模块设计。

核心原则：

```text
功能模块不是越多越好，而是要服务一期原型最小闭环。
```

## 一、本节目标

完成 2-3 后，我们已经知道：

```text
谁在用系统
在什么场景下用
当前怎么做
痛点是什么
系统应该支持什么
```

2-4 要进一步回答：

```text
为了支持这些场景，系统到底需要哪些功能？
哪些功能一期必须做？
哪些功能可以后续扩展？
哪些功能暂时不要做？
```

这一步的输出会直接影响后续：

```text
页面清单
页面字段
接口设计
Vue3 组件生成
测试用例
演示脚本
```

## 二、功能模块设计的基本思路

功能模块设计不要从“系统应该有什么菜单”开始，而要从场景任务反推。

推荐顺序：

```text
P0/P1 业务场景
  -> 用户任务
  -> 系统动作
  -> 输入数据
  -> 输出结果
  -> 功能模块
  -> 页面与接口
```

例如：

```text
客户需求初读
  -> 输入需求文件路径
  -> 系统调用后端生成报告
  -> 返回成功或错误结果
  -> 需要“需求分析生成”功能模块
```

## 三、功能模块分级标准

为了避免原型范围失控，每个功能都要分级。

| 分级 | 含义 | 判断标准 |
|---|---|---|
| 必须做 | 一期原型最小闭环必需 | 没有它，主流程跑不通 |
| 可以做 | 增强演示效果 | 有它更完整，但没有也能演示 |
| 暂不做 | 后续扩展能力 | 当前实现成本高、风险大或与主线关系弱 |

建议一期原型只保留：

```text
必须做功能 + 少量可以做功能
```

不要把所有客户想法都塞进第一版原型。

| 范围类型 | 内容 | 说明 |
|---|---|---|
| 一期必须包含 | 工作台首页、客户需求分析页、API 调用、结果展示 | 保证最小闭环 |

| 一期不包含 | 用户登录、权限管理、文件上传、多格式解析 | 避免复杂度过高 |

## 八、示例：客户需求分析助手

以下示例基于第一阶段已有 FastAPI 能力。

### 1. 功能模块总表

| 优先级 | 功能模块 | 功能说明 | 服务场景 | 使用角色 | 业务价值 |
|---|---|---|---|---|---|
| 必须做 | 工作台导航 | 展示原型工具入口和系统能力 | 原型使用入口 | 售前/FDE/项目负责人 | 让用户快速理解系统能做什么 |
| 必须做 | 需求分析参数输入 | 输入需求文件路径、输出路径、AI 建议开关 | 客户需求初读 | 售前/FDE | 把分析任务参数化 |
| 必须做 | 需求报告生成 | 调用后端生成结构化需求分析报告 | 客户需求初读 | 售前/FDE | 降低人工整理成本 |
| 必须做 | 执行结果展示 | 展示成功、失败、输出路径和错误码 | 客户需求初读 | 售前/FDE | 让用户明确任务结果 |
| 可以做 | 后端健康检查 | 检查 FastAPI 服务是否可用 | 原型演示准备 | FDE | 降低联调排错成本 |
| 可以做 | Prompt 模板展示 | 展示需求拆解、页面生成、联调 Prompt | 原型开发准备 | FDE | 沉淀方法资产 |
| 暂不做 | 文件上传 | 上传 Markdown/Word/PDF 并自动解析 | 客户需求初读 | 售前/FDE | 第二阶段先复用文件路径，避免解析复杂度 |
| 暂不做 | 用户登录权限 | 区分不同用户角色权限 | 系统管理 | 管理员 | 当前为本地原型，暂不引入 |

### 2. 功能点明细表

| 功能模块 | 功能点 | 用户操作 | 系统动作 | 输入 | 输出 | 异常情况 |
|---|---|---|---|---|---|---|
| 工作台导航 | 查看功能入口 | 打开首页 | 展示功能卡片和菜单 | 无 | 功能入口列表 | 无 |
| 需求分析参数输入 | 输入文件路径 | 填写 input_path | 绑定表单状态 | input_path | 表单数据 | 路径为空 |
| 需求分析参数输入 | 设置输出路径 | 填写 output_path | 绑定表单状态 | output_path | 表单数据 | 路径为空 |
| 需求分析参数输入 | 启用 AI 建议 | 打开开关 | 设置 enable_ai_advice | true/false | 表单数据 | API Key 未配置 |
| 需求报告生成 | 提交分析任务 | 点击生成报告 | 调用 `/requirement/summary` | input_path/output_path/enable_ai_advice | API 响应 | 文件不存在、LLM 错误、后端异常 |
| 执行结果展示 | 查看成功结果 | 等待接口返回 | 展示 success/message/output_path | API response | 成功提示 | 响应为空 |

| 功能模块 | 优先级 | 进入一期原型 | 原因 | 后续扩展方向 |
|---|---|---|---|---|
| 工作台导航 | 必须做 | 是 | 用户需要入口理解系统能力 | 增加最近任务和状态统计 |

### 6. 一期原型范围表

| 范围类型 | 内容 | 说明 |
|---|---|---|
| 一期必须包含 | 工作台首页、客户需求分析页、需求报告生成、结果展示 | 保证最小闭环可演示 |

## 九、功能模块设计质量检查

完成后，用下面清单检查。

| 检查项 | 合格标准 |
|---|---|
| 是否来自场景 | 每个功能都能追溯到 P0/P1 场景 |
| 是否有优先级 | 每个功能都有必须做、可以做或暂不做 |
| 是否能形成闭环 | 一期功能能跑通输入、处理、输出 |
| 是否避免堆功能 | 没有把二期、三期功能塞进一期 |
| 是否能指导页面 | 每个功能能映射到页面或页面区域 |
| 是否能指导接口 | 需要后端支持的功能有接口映射 |
| 是否能用于演示 | 能支撑 3 分钟讲清楚价值 |
