# 2-33 场景结果写回并传递到功能设计

## 本节目标

本节在 2-32 的“需求结果写回并传递到场景识别”基础上，继续打通下一段链路：

将场景识别中的当前选中场景写回统一项目上下文，并传给功能设计工作台展示。

当前仍然不调用 DeepSeek，也不调用 FastAPI，不引入新的 npm 依赖。本节继续采用“小步闭环”方式，验证“需求 -> 场景 -> 功能”的上下文传递可行性。

## 本节价值

2-32 已经让场景识别页能看到来自需求输入的上一步结果。

本节进一步实现：

1. 场景识别页将“当前选中场景”确认写回上下文
2. 一键进入功能设计页
3. 功能设计页读取这份场景上下文并展示“来源场景确认”

这一步会让第一 Tab 更像真实工作流，而不是多个独立页面。

## 当前交互规则

- 场景识别页可将当前选中场景写入项目上下文。
- 点击“写入场景并进入功能设计”后，自动进入功能设计页。
- 功能设计页展示来自场景识别阶段的确认信息。
- 当前仍然使用 mock 结果，后续再替换为真实接口返回。

## 后续扩展方向

后续可以继续扩展：

- 将功能设计页当前模块选择写回上下文并传给页面设计。
- 在上下文中记录每一步确认时间、确认人和版本号。
- 接入后端后，把上下文写回改为真实项目状态持久化。
