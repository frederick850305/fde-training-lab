# 2-2 客户需求快速拆解模板

本文件用于把客户原始需求快速转成适合原型开发的结构化需求。

使用原则：

```text
先拆需求，不急着写代码。
先确认业务目标和演示重点，再生成页面。
```

## 一、本节目标

拿到客户需求后，不要直接对 Codex/Copilot 说：

```text
帮我做一个系统原型。
```

更推荐先让 AI 完成需求拆解：

```text
客户原始材料
  -> 业务背景
  -> 客户痛点
  -> 业务目标
  -> 用户角色
  -> 核心场景
  -> 功能模块
  -> 页面建议
  -> 原型优先级
  -> 待确认问题
```

这一步的作用是让后续 Vue3 页面生成更准确，避免 AI 生成“看起来很漂亮、但不贴业务”的页面。

## 二、适用输入材料

这个模板适合处理以下材料：

| 输入类型 | 示例 |
|---|---|
| 客户访谈纪要 | 客户口头描述、会议记录、微信沟通内容 |
| 招标文件片段 | 建设目标、功能要求、技术要求、评分项 |
| Word/PPT 方案 | 系统建设方案、项目建议书、汇报材料 |
| 业务痛点描述 | 当前人工流程、Excel 台账、微信群调度、系统割裂问题 |
| 竞品或参考系统 | 客户说“想做一个类似某系统的能力” |

## 三、需求拆解输出结构

每次需求拆解建议固定输出以下 9 个部分。

| 序号 | 输出项 | 说明 | 用途 |
|---:|---|---|---|
| 1 | 业务背景 | 客户处于什么行业、什么业务环节、什么管理场景 | 判断页面风格和业务语言 |
| 2 | 客户痛点 | 当前低效、风险、成本、协同、数据问题 | 决定原型演示要打中什么问题 |
| 3 | 业务目标 | 客户希望通过系统达到什么效果 | 决定原型主线 |
| 4 | 用户角色 | 谁会使用系统，谁看结果，谁管理流程 | 决定页面和权限 |
| 5 | 核心业务场景 | 系统在哪些工作场景中被使用 | 决定功能模块 |
| 6 | 功能模块建议 | 原型应包含哪些模块 | 决定开发范围 |
| 7 | 页面清单建议 | 前端需要哪些页面 | 决定 Vue3 页面结构 |
| 8 | 原型开发优先级 | 哪些先做，哪些后做 | 控制原型范围 |
| 9 | 待确认问题 | 哪些信息还不清楚 | 防止假设过多 |

## 五、面向 FDE 原型的增强 Prompt


## 六、需求拆解结果验收清单

完成需求拆解后，用下面清单检查质量。

| 检查项 | 合格标准 |
|---|---|
| 是否说清业务背景 | 能看出客户行业、业务环节和管理对象 |
| 是否提炼真实痛点 | 不是泛泛写“效率低”，而是能说清哪里低效 |
| 是否识别用户角色 | 至少区分操作人员、管理人员、决策人员 |
| 是否有核心场景 | 场景能对应到真实工作流 |
| 是否分清优先级 | 明确哪些必须做，哪些可以后做 |
| 是否能指导页面设计 | 页面清单不是空泛模块名，而是能继续设计字段和按钮 |
| 是否标注不确定问题 | 不清楚的信息进入待确认问题，不强行编造 |
| 是否适合原型演示 | 能形成一个 3 分钟讲得清楚的演示主线 |

## 七、示例：客户需求分析助手

以下是基于第一阶段项目能力整理出的示例需求拆解。

### 1. 业务背景

FDE 在售前和原型开发过程中，经常需要快速阅读客户需求材料，并提炼业务目标、用户角色、核心需求和初步功能模块。

### 2. 客户痛点

| 痛点 | 说明 |
|---|---|
| 需求材料非结构化 | 客户需求可能来自 Markdown、Word、会议纪要或招标片段 |
| 人工整理耗时 | 售前人员需要反复提炼业务目标和功能范围 |
| 输出格式不统一 | 不同人员整理出来的需求摘要格式不一致 |
| 原型设计起步慢 | 缺少从需求到页面设计的标准过渡 |

### 3. 业务目标

```text
构建一个客户需求分析助手，让 FDE 能快速把客户原始需求转成结构化分析报告，并作为后续原型设计输入。
```

### 4. 用户角色

| 角色 | 主要关注点 | 可能使用的页面 |
|---|---|---|
| 售前解决方案工程师 | 快速理解客户需求，形成方案初稿 | 客户需求分析页、报告结果页 |
| FDE 工程师 | 将需求转成功能和接口设计 | 需求分析页、Prompt 模板页 |
| 项目负责人 | 判断是否值得继续投入原型 | 工作台首页、报告结果页 |

### 5. 核心业务场景

| 场景 | 触发条件 | 当前做法 | 系统化改进方向 |
|---|---|---|---|
| 客户需求初读 | 拿到客户需求文档后 | 人工阅读并摘录 | 自动生成结构化摘要 |
| 原型开发准备 | 准备生成页面前 | 临时梳理功能 | 输出功能模块和页面建议 |
| 售前汇报准备 | 需要向团队说明需求 | 手工整理材料 | 生成统一格式报告 |

### 6. 功能模块建议

| 优先级 | 功能模块 | 功能说明 | 对应业务价值 |
|---|---|---|---|
| 必须做 | 需求文件路径输入 | 输入待分析的需求文件路径 | 支持快速指定材料 |
| 必须做 | 结构化报告生成 | 调用后端生成需求分析报告 | 降低人工整理成本 |
| 必须做 | 执行结果展示 | 展示生成状态、输出路径和提示信息 | 方便确认执行结果 |
| 可以做 | AI 辅助建议 | 调用 DeepSeek 生成实施建议 | 增强方案初稿质量 |
| 可以做 | 报告内容预览 | 在页面中直接预览 Markdown 内容 | 提升演示体验 |
| 暂不做 | 在线上传 Word/PDF | 上传并解析多格式文档 | 第二阶段先复用文件路径 |

### 7. 页面清单建议

| 页面名称 | 页面目标 | 主要用户 | 对应功能模块 | 原型优先级 |
|---|---|---|---|---|
| 工作台首页 | 展示原型工具入口和能力概览 | 全部用户 | 功能导航 | 必须做 |
| 客户需求分析页 | 输入路径并生成报告 | 售前/FDE | 需求分析 | 必须做 |
| 报告结果区 | 展示执行结果和错误提示 | 售前/FDE | 结果展示 | 必须做 |
| Prompt 模板页 | 展示可复用 Prompt | FDE | 方法资产 | 可以做 |

### 8. 原型开发优先级

一期原型建议只做一个闭环：

```text
输入需求文件路径
  -> 点击生成报告
  -> 调用 FastAPI
  -> 返回成功或失败信息
  -> 页面展示结果
```

原因：

```text
这个闭环最短，能够验证 Vue3 页面、FastAPI 接口、错误响应和原型演示流程。
```

### 9. 待确认问题

| 问题 | 为什么需要确认 | 如果暂时无法确认，原型中的默认假设 |
|---|---|---|
| 是否需要文件上传 | 决定页面是路径输入还是上传组件 | 第二阶段先使用文件路径 |
| 是否需要直接预览报告内容 | 决定是否新增读取报告接口 | 先展示输出路径，后续再扩展 |
| 是否默认启用 AI 建议 | 涉及 DeepSeek Key 和调用费用 | 默认关闭，提供开关 |
