# 2-30 原型工厂操作台总览

## 本节目标

本节在 2-29 的“工作台一级 Tab 结构重组”基础上，继续升级第一个 Tab，使其从单纯的“方法展示区”转向更明确的“原型工厂操作台总览”。

当前仍然不调用 DeepSeek，也不调用 FastAPI，不引入新的 npm 依赖。本节重点不是生成新的业务原型页，而是把“这套系统未来如何用于真实客户需求分析与原型生成”在前台界面中表达清楚。

## 本节价值

前面已经完成：

- 需求输入、场景识别、功能设计、页面设计、交互设计、API 契约、前端原型建议等单点工作台
- P0 业务原型页面
- 一级 Tab 和业务原型内部导航

但第一个 Tab 仍然容易让人产生困惑：

- 当前页面哪些是演示型 mock 工作台？
- 未来接入 DeepSeek/FastAPI 后，这些工作台如何串成真实链路？
- 每一步的输入、输出和下一步是什么？

本节通过新增“原型工厂操作台总览”页，明确展示：

1. 当前阶段有哪些步骤已经有前端承载
2. 每一步的输入、输出和当前实现模式
3. 后续如何从 mock 演进到真实执行链路

## 页面能力

本节新增的“原型工厂操作台总览”包含：

1. 当前阶段摘要
2. 原型工厂步骤链路总览
3. 每一步的输入、输出、当前模式和后续目标
4. 一键进入对应工作台
5. 从 mock 演示到真实实战的演进提示

## 当前交互规则

- 第一个一级 Tab 默认进入“原型工厂操作台总览”。
- 点击步骤卡片中的按钮，进入对应工作台。
- 点击一级 Tab“原型工厂方法”时，默认回到总览页。
- 当前仍然使用本地 mock 数据，不做真实调用。

## 后续扩展方向

后续可以继续扩展：

- 将步骤状态接入后端任务执行状态。
- 为每一步增加“执行 / 重跑 / 审核通过 / 退回修改”动作。
- 将输入输出结果保存为项目上下文。
- 接入 DeepSeek/FastAPI 后，将当前 mock 说明替换为真实执行结果。

## 验收方式

执行前端构建：

```bash
cd frontend
npm run build
```

浏览器打开页面后确认：

- 第一个 Tab 默认展示“原型工厂操作台总览”。
- 页面能说明每一步的输入、输出、当前实现模式和未来目标。
- 点击步骤或能力入口按钮后，可以进入对应工作台。
- 第一个 Tab 的结构比之前更像“未来可实战的操作台主页”。
- 第二阶段练习进度更新为 `2-30：原型工厂操作台总览`。
