# 2-1 AI 原型开发标准流程

本文件是第二阶段的第一份方法资产，用于沉淀一套可复用的 AI 原型开发流程。

目标不是一次性让 AI 直接生成完整系统，而是建立稳定的工作节奏：

```text
客户需求
  -> 需求梳理
  -> 业务场景识别
  -> 功能模块设计
  -> 页面清单设计
  -> 页面字段与交互设计
  -> API 契约设计
  -> Prompt 准备
  -> Vue3 前端原型生成
  -> FastAPI 后端联调
  -> 原型验收
  -> 演示包装
```

## 一、阶段定位

第二阶段名称：

```text
基于 VSCode + Codex/Copilot + Vue3 的 AI 原型工厂训练
```

第二阶段不以传统前端学习为主，而是训练一种更贴近 FDE 工作场景的方法：

```text
拿到客户需求后，能够借助 AI 快速完成需求拆解、功能设计、页面设计、Prompt 编写和前端原型生成。
```

核心产出包括两类：

| 类型 | 内容 |
|---|---|
| 原型系统 | 基于 Vue3 + FastAPI 的 FDE Prototype Factory Web Console |
| 方法资产 | 需求梳理、功能设计、页面设计、接口契约、Prompt、联调、验收和演示模板 |

## 二、标准流程总览

| 步骤 | 名称 | 你要做什么 | AI 帮你做什么 | 输出物 |
|---|---|---|---|---|
| 1 | 输入客户需求 | 粘贴客户原始需求、会议纪要、招标文件片段 | 提取业务目标、痛点、约束 | 原始需求材料 |
| 2 | 需求结构化 | 判断客户到底要解决什么问题 | 输出业务背景、痛点、目标、约束 | 需求分析表 |
| 3 | 场景识别 | 明确用户在什么场景下使用系统 | 梳理业务场景、用户角色、操作流程 | 场景角色表 |
| 4 | 功能设计 | 把需求转成功能模块 | 设计模块、功能点、优先级 | 功能模块表 |
| 5 | 页面设计 | 把功能模块转成页面 | 设计页面清单、导航结构 | 页面清单 |
| 6 | 交互设计 | 明确每个页面如何操作 | 设计字段、按钮、状态、提示 | 页面交互说明 |
| 7 | API 契约 | 明确前后端如何通信 | 设计请求参数、响应结构、错误结构 | API 契约文档 |
| 8 | Prompt 准备 | 把页面说明转成代码生成提示词 | 生成 Vue3 页面 Prompt | Prompt 模板 |
| 9 | 前端生成 | 在 VSCode/Codex/Copilot 中生成代码 | 生成 Vue3 页面和组件 | 前端代码 |
| 10 | 接口联调 | 启动 FastAPI 和 Vue3 | 修复调用、跨域、参数、响应适配问题 | 可运行原型 |
| 11 | 原型验收 | 按业务、功能、体验检查 | 生成验收清单和修改建议 | 验收记录 |
| 12 | 演示包装 | 面向客户或领导演示 | 生成 3 分钟演示脚本 | 演示材料 |

## 三、VSCode 中的推荐工作节奏

在 VSCode 中使用 Codex/Copilot 做原型时，建议按以下节奏推进。

### 1. 先建上下文，不直接写代码

先把客户需求整理成 Markdown，并要求 AI 只做分析，不写代码。

推荐输出：

```text
需求分析表
场景角色表
功能模块表
页面清单
接口契约
```

这样可以避免 AI 一开始就生成一个看起来漂亮、但业务不准的页面。

### 2. 再让 AI 生成页面

当页面清单和页面交互说明清楚后，再让 AI 生成 Vue3 页面。

推荐做法：

```text
一次只生成一个页面
一次只改一个明确问题
每次修改前先说明保留哪些现有逻辑
```

### 3. 小步联调

不要等所有页面都写完再联调。

推荐顺序：

```text
先调 /health
再调一个简单 POST 接口
再调复杂业务接口
最后统一错误处理和 UI 状态
```

### 4. 保留 Prompt 资产

每次效果好的 Prompt 都要沉淀下来。

建议分类保存：

```text
需求梳理 Prompt
功能设计 Prompt
页面生成 Prompt
组件拆分 Prompt
API 联调 Prompt
UI 优化 Prompt
错误修复 Prompt
验收检查 Prompt
演示脚本 Prompt
```

## 四、基础 Prompt 模板

### 1. 需求梳理 Prompt

```text
我正在做一个售前原型设计。请不要直接写代码。

请先基于下面的客户需求，按以下结构帮我完成原型开发前的需求梳理：

1. 业务背景
2. 客户痛点
3. 业务目标
4. 用户角色
5. 核心业务场景
6. 功能模块建议
7. 页面清单建议
8. 需要确认的问题
9. 原型开发优先级

客户需求如下：
"""
在这里粘贴客户需求
"""
```

### 2. 功能设计 Prompt

```text
请基于上面的需求分析结果，帮我继续拆解原型功能模块。

要求：
1. 按模块输出
2. 每个模块包含功能点、使用角色、输入、输出、优先级
3. 区分一期原型必须做和后续可扩展
4. 输出为 Markdown 表格
5. 不要写代码
```

### 3. 页面设计 Prompt

```text
请基于功能模块表，帮我设计 Vue3 前端原型的页面清单。

要求输出：
1. 页面名称
2. 页面目标
3. 主要用户
4. 页面字段
5. 主要按钮
6. 页面状态，包括默认、loading、成功、失败、空数据
7. 对应后端接口
8. 页面优先级

请用 Markdown 表格输出。
```

### 4. Vue3 页面生成 Prompt

```text
请基于下面的页面设计说明，生成 Vue3 + Vite 的单文件组件。

技术要求：
1. 使用 Vue3 script setup
2. 使用原生 CSS 或 scoped CSS
3. 不引入复杂 UI 组件库
4. 页面要适合售前原型演示
5. 包含表单输入、按钮、loading、成功、失败、空状态
6. 代码结构清晰，变量命名直观

页面设计说明如下：
"""
粘贴页面设计说明
"""
```

### 5. API 联调 Prompt

```text
请基于下面的 FastAPI 接口契约，帮我生成 Vue3 前端的 axios 调用代码。

要求：
1. 封装为独立 api 文件
2. 包含 baseURL 配置
3. 正确处理 success/message/data/error_code 结构
4. 适配 loading、成功、失败三类状态
5. 不要修改无关页面代码

接口契约如下：
"""
粘贴接口契约
"""
```

### 6. 排错 Prompt

```text
下面是我在 Vue3 + FastAPI 联调时遇到的问题。

请你先判断问题类型，再给出最小修改方案。

要求：
1. 先说明可能原因
2. 再说明需要检查的文件
3. 最后给出建议修改
4. 不要大范围重写已有代码

错误信息：
"""
粘贴浏览器控制台、终端或接口返回错误
"""

相关代码：
"""
粘贴相关代码片段
"""
```

## 六、第二阶段验收标准

完成第二阶段后，需要满足以下标准：

| 验收项 | 标准 |
|---|---|
| 流程资产 | 形成从需求到原型的标准流程文档 |
| Prompt 资产 | 沉淀需求、功能、页面、Vue3、联调、排错和验收 Prompt |
| 前端项目 | 有可运行的 Vue3 + Vite 前端工程 |
| 后端联调 | 能调用第一阶段 FastAPI 接口 |
| 状态处理 | 具备 loading、成功、失败、空状态 |
| 错误处理 | 能展示统一错误响应中的 message 和 error_code |
| 页面质量 | 页面适合售前演示，结构清晰，业务表达准确 |
| 文档说明 | README 说明如何启动、联调和演示 |
| 演示材料 | 有 3 分钟演示脚本 |
| 方法复用 | 拿到新客户需求后，可以按模板重复生成新原型 |

## 七、本节结论

第二阶段的核心不是学会某个前端语法点，而是建立一个可复用的 AI 原型生产流程：

```text
先用 AI 帮你想清楚需求和页面
再用 AI 帮你生成 Vue3 代码
最后用 AI 帮你联调、排错、验收和包装演示
```

这套流程后续会继续拆成模板、Prompt 和代码实践，逐步变成你自己的 FDE 原型工厂。
