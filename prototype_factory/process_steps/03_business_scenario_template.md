# 2-3 业务场景与用户角色识别模板

本文件用于把客户需求中的“用户角色”和“业务场景”进一步拆细，为后续功能设计、页面设计、演示脚本提供依据。

核心原则：

```text
先搞清楚谁在用、什么时候用、为了解决什么问题，再设计页面。
```

## 一、本节目标

完成 2-2 的客户需求拆解后，通常已经有了初步用户角色和核心场景。

但这些信息往往还比较粗，例如：

```text
用户角色：管理员、业务人员、领导
业务场景：数据分析、报告生成、系统管理
```

这还不够指导原型开发。

2-3 要进一步拆成：

```text
角色
  -> 关注目标
  -> 日常任务
  -> 使用场景
  -> 操作步骤
  -> 需要看的页面
  -> 需要系统给出的结果
```

这样后续生成 Vue3 页面时，页面才会更贴近业务，而不是只有空泛的卡片和按钮。

## 二、为什么 FDE 要先识别角色和场景

FDE 做原型不是为了“展示系统有多少功能”，而是为了让客户看到：

```text
这个系统能不能解决我每天真实遇到的问题。
```

所以原型的设计顺序应该是：

```text
角色
  -> 场景
  -> 任务
  -> 页面
  -> 功能
  -> 接口
```

而不是：

```text
功能模块
  -> 随便生成页面
```

## 三、用户角色识别模板

建议用下面的表格识别用户角色。

| 角色 | 角色类型 | 主要目标 | 日常任务 | 关注指标 | 使用频率 | 典型页面 |
|---|---|---|---|---|---|---|
| 一线操作人员 | 执行者 | 完成具体业务操作 | 填报、查询、确认、处理异常 | 任务状态、提醒、处理结果 | 高频 | 操作页、详情页 |
| 业务管理人员 | 管理者 | 掌握业务进展和风险 | 审核、分派、跟踪、协调 | 进度、风险、效率、完成率 | 高频 | 工作台、列表页、看板页 |
| 决策领导 | 决策者 | 看趋势、风险和价值 | 查看汇总、听取汇报、做决策 | 总览指标、趋势、异常、收益 | 中低频 | 首页、驾驶舱、报表页 |
| 系统管理员 | 支撑者 | 保障系统可用和规则可配 | 用户配置、字典配置、权限维护 | 系统状态、配置完整性 | 中频 | 系统配置页 |

角色类型建议固定为：

| 类型 | 说明 |
|---|---|
| 执行者 | 直接处理业务任务的人 |
| 管理者 | 分配、审核、协调、跟踪任务的人 |
| 决策者 | 看汇总结果、风险和价值的人 |
| 支撑者 | 维护系统、配置规则、管理权限的人 |

## 四、业务场景识别模板

业务场景要尽量描述成真实工作片段。

不要只写：

```text
需求分析场景
```

更推荐写：

```text
售前工程师拿到客户需求文档后，需要在 30 分钟内整理出业务目标、核心需求和原型页面建议。
```

建议使用下面的模板：

| 场景名称 | 触发时机 | 参与角色 | 当前做法 | 当前痛点 | 系统支持方式 | 预期输出 |
|---|---|---|---|---|---|---|
| 客户需求初读 | 拿到需求文档后 | 售前/FDE | 人工阅读摘录 | 耗时、格式不统一 | 自动结构化分析 | 需求分析报告 |

## 五、任务流程拆解模板

每个核心场景都建议继续拆成任务流程。

| 步骤 | 用户动作 | 系统动作 | 输入信息 | 输出结果 | 异常情况 |
|---:|---|---|---|---|---|
| 1 | 输入需求文件路径 | 校验路径是否为空 | 文件路径 | 校验结果 | 路径为空 |
| 2 | 点击生成报告 | 调用后端接口 | input_path/output_path | 接口响应 | 文件不存在 |
| 3 | 查看执行结果 | 展示成功或失败信息 | API response | 输出路径或错误提示 | 后端异常 |

这个表后续可以直接转换成：

```text
页面字段
按钮
loading 状态
成功状态
失败状态
API 契约
测试用例
```

## 六、场景优先级判断

不是所有场景都适合第一版原型。

FDE 原型优先选择以下场景：

| 判断维度 | 优先选择 |
|---|---|
| 是否高频 | 高频操作优先 |
| 是否痛点明确 | 客户明确抱怨的问题优先 |
| 是否容易演示 | 3 分钟内能讲清楚优先 |
| 是否能形成闭环 | 有输入、处理、输出优先 |
| 是否风险可控 | 不涉及复杂权限和真实生产控制优先 |
| 是否能复用已有能力 | 能复用现有 FastAPI/service 优先 |

场景优先级建议分为：

| 优先级 | 说明 |
|---|---|
| P0 | 一期原型必须演示，构成最小闭环 |
| P1 | 一期可以做，增强演示完整度 |
| P2 | 后续扩展，先保留设计 |
| P3 | 暂不进入原型，避免范围失控 |

## 七、角色与页面映射模板

识别角色和场景后，要进一步映射到页面。

| 角色 | 核心场景 | 需要页面 | 页面上最重要的信息 | 页面上最重要的操作 |
|---|---|---|---|---|
| 售前/FDE | 客户需求初读 | 客户需求分析页 | 输入文件路径、生成状态、输出路径 | 生成报告 |
| 项目负责人 | 原型评估 | 工作台首页 | 已有能力、最近输出、系统状态 | 查看结果 |

这个表可以帮助你判断：

```text
页面是否真的服务角色
页面上是否放了正确的信息
按钮是否对应真实任务
```

## 十、示例：客户需求分析助手

### 1. 用户角色识别

| 角色 | 角色类型 | 主要目标 | 日常任务 | 关注指标 | 使用频率 | 典型页面 |
|---|---|---|---|---|---|---|
| 售前解决方案工程师 | 执行者 | 快速整理客户需求 | 阅读材料、提炼需求、准备方案 | 整理耗时、输出完整性 | 高频 | 客户需求分析页 |
| FDE 工程师 | 执行者 | 将需求转成原型输入 | 分析需求、设计页面、准备 Prompt | 功能清晰度、页面可生成性 | 高频 | Prompt 模板页、需求分析页 |
| 项目负责人 | 管理者 | 判断原型是否值得推进 | 查看分析结果、确认范围 | 原型范围、投入产出 | 中频 | 工作台首页、报告结果页 |
| 业务领导 | 决策者 | 判断系统价值 | 查看演示和摘要 | 效率提升、交付风险 | 低频 | 演示首页、报告摘要区 |

### 2. 业务场景识别

| 场景名称 | 触发时机 | 参与角色 | 当前做法 | 当前痛点 | 系统支持方式 | 预期输出 |
|---|---|---|---|---|---|---|
| 客户需求初读 | 刚拿到需求材料 | 售前/FDE | 人工阅读、手工摘录 | 耗时、格式不统一 | 自动生成结构化报告 | 需求分析报告 |
| 原型开发准备 | 准备让 AI 生成页面前 | FDE | 临时整理功能和页面 | 容易遗漏字段和状态 | 输出页面建议和 Prompt 输入 | 页面设计输入 |

### 3. 核心场景任务流程

| 客户需求初读 | 4 | 查看结果 | 展示成功/失败提示 | response | 输出路径或错误说明 | 响应结构异常 |

## 十一、输出质量检查

完成角色和场景识别后，用下面清单检查。

| 检查项 | 合格标准 |
|---|---|
| 角色是否真实 | 能对应到客户组织中的真实岗位或职责 |
| 场景是否具体 | 能说清触发时机和当前做法 |
| 痛点是否落地 | 不是泛泛写效率低，而是能说清哪个动作低效 |
| 流程是否闭环 | 至少有输入、处理、输出 |
| 页面映射是否明确 | 能看出每个角色需要哪个页面 |
| 优先级是否克制 | P0 不超过 1-2 个核心场景 |
| 是否服务演示 | 能支撑 3 分钟讲清楚一条业务主线 |
