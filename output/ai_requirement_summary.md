# 客户需求结构化分析报告

## 一、业务目标

- 快速检索历史故障案例
- 自动归纳故障原因
- 生成标准化处置建议
- 沉淀专家经验

## 二、用户角色

- 现场工程师
- 设备管理员
- 专家审核人员
- 系统管理员

## 三、核心需求

- 支持导入故障案例
- 支持按设备类型检索
- 支持生成处理建议
- 支持专家审核和发布

## 四、初步功能模块建议

| 模块 | 建议 |
|---|---|
| 知识导入模块 | 支持故障案例、手册、专家经验等资料录入 |
| 知识检索模块 | 支持按设备类型、故障现象、关键词进行查询 |
| 智能建议模块 | 根据故障现象生成标准化处置建议 |
| 专家审核模块 | 支持专家复核、发布、版本留痕和回滚 |

## 五、初步分析结论

该需求具备典型的“知识沉淀 + 智能检索 + 专家审核 + 标准处置建议”特征，适合先建设一个轻量级故障知识助手 MVP。第一阶段不宜追求大而全，建议优先实现资料导入、基础检索、处置建议生成和专家审核闭环。

## 六、AI 辅助实施建议

### 1. 建设重点判断

- **案例数据标准化是基础**：故障案例的快速检索和自动归纳高度依赖结构化数据（如设备型号、故障现象、原因分类、处置步骤）。若原始案例多为非结构化文本，需优先定义统一的数据模板，否则后续检索准确率和归纳效果将严重受限。
- **检索性能与体验优先于AI深度能力**：现场工程师的核心痛点在于“快”。应优先保证全字段模糊检索、设备类型分层筛选、历史相似案例秒级返回，而非急于实现高复杂度的自动原因归纳——后者对数据量和模型训练要求较高，可放在第二阶段。
- **专家审核机制需闭环且低阻滞**：用户明确要求“专家审核和发布”，这直接决定知识库的权威性与可用性。需设计轻量审核流（如单节点+加签），避免因流程过重导致案例积压、专家不愿参与。

### 2. MVP 优先级建议

| 优先级 | 建设内容 | 说明 |
|--------|----------|------|
| **高** | 故障案例导入（手动/批量Excel/API） | 支持字段映射、必填校验、附件上传，覆盖现场工程师录入与管理员批量初始化 |
| **高** | 按设备类型、关键字、时间范围检索 | 基于Elasticsearch或关系库全文索引，返回案例标题、故障现象、处置摘要，支持模糊匹配 |
| **高** | 简单处理建议生成 | 基于检索到的历史案例，抽取“处置步骤”字段拼接展示，不依赖NLP，可人工补充模板化建议 |
| **高** | 专家审核与发布流程 | 待审核列表、审核通过/驳回/修订、已发布案例锁定编辑；支持通知提醒 |
| **中** | 自动归纳故障原因 | 初期仅实现基于规则（如高频关键词聚类），待数据量>500条后再引入分类或摘要模型，避免空转 |
| **中** | 设备管理员视图 | 设备台帐关联、案例统计（按设备、原因、时间）、未处理案例看板 |
| **低** | 高级语义搜索（同义词、联想） | 可延迟至二期；一期采用标签+设备类型+关键词即可基本满足需求 |
| **低** | 移动端适配 | 若现场工程师使用手机较多，可考虑适配H5，但非MVP必选 |

### 3. 主要风险点

1. **案例数据质量低导致AI归纳失效**：客户现有案例可能缺失关键字段（如原因、处置步骤），或描述口语化严重。直接投入NLP建模，输出结果可能被用户否定，造成信任危机。
2. **专家审核成为流程瓶颈**：若专家数量少或业务繁忙，审核积压会使“标准化处置建议”迟迟无法发布，现场工程师将失去使用动力。
3. **检索结果重复/无关**：不同时间录入的同一故障可能描述差异大，导致检索召回率低或准确率差，用户反复搜不到满意案例。
4. **用户不愿主动沉淀经验**：现场工程师缺乏提交案例的动力，系统初期内容空洞，形成“冷启动”困境。需在考核或激励上做配套制度设计，而非仅依赖技术手段。
5. **与其他系统（工单系统、设备管理系统）集成成本被低估**：若设备台账、人员组织、工单数据需实时同步，接口协议、字段映射、历史数据迁移的工作量可能远超预期。

### 4. 乙方实施建议

- **需求澄清**：
  - 明确“自动归纳故障原因”的预期准确率（建议初期定义为“候选推荐”，而非自动填写），避免过度承诺。
  - 与客户共同梳理3-5种典型设备故障场景的案例样本，现场演示检索与建议生成的效果，对齐验收标准。

- **范围控制**：
  - 将“高级AI分析”单独作为二期规划，一期聚焦在数据标准化、搜索与审核流程上。
  - 若客户坚持纳入，建议限定为基于规则（如关键字→原因模板），并设置数据量阈值（如不足200条暂不激活）。

- **数据准备**：
  - 要求客户提供至少50-100条历史典型案例作为初始语料，乙方负责清洗、补全字段、去重。
  - 设计数据模板时，预留“原因分类”和“处置步骤”为必填字段，并推荐使用下拉列表以降低录入难度。

- **接口集成**：
  - 优先对接客户现有工单系统（如Zendesk、Jira、泛微），从工单中自动抽取“故障现象+处理方法”生成案例草稿，减少人工录入。
  - 设备台账通过轻量API或定期CSV同步，避免实时接口带来的性能与权限风险。

- **验收标准**：
  - 检索：在1000条测试数据中，按设备类型+关键词组合查询，返回结果前5条准确率≥80%，响应时间<1秒。
  - 审核流程：从提交到发布，正常流程（单节点审核）完成时间≤2小时（排除人为等待）。
  - 处理建议生成：针对测试集的10个已知故障，系统推荐的处置建议与人工撰写的内容匹配率（关键步骤）≥70%。
